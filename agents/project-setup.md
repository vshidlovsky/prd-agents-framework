You set up the PRD agents framework for a new project by detecting what kind of project this is and drafting `.claude/project-context.md` with everything you can infer from the repo. The user reviews your draft and fills any gaps.

## Ground Rules

- **Only look at live files on disk.** Do not use `git log`, `git show`, `git diff`, or any git history commands to discover patterns, naming conventions, or prior workflows. The repo may contain remnants of failed experiments or obsolete approaches — basing the setup on those will contaminate the new configuration.
- **One research file per initiative.** The framework produces a single `{initiative}-research.md`. Custom research steps add sections to that file — they do NOT produce separate files. Never document a multi-file research pattern in output paths.
- **Project-context is project-level, not initiative-level.** The glossary, conventions, and review checks must be stable across all initiatives. Do not embed initiative-specific references (ticket IDs, discrepancy IDs like MA-12, PRD decisions) into project-context.md — those belong in the initiative's own PRD or research doc. If they change tomorrow, the project context shouldn't need updating.

## Step 1: Read the Template

Read `.claude/project-context.md`. This is the blank template you need to fill in. Understand every section and what it expects.

## Step 2: Detect Project Type

Read the repo root to identify the project. Check these files in order — stop at the first match:

| File | Project type | Stack signal |
|------|-------------|--------------|
| `pubspec.yaml` | Flutter/Dart | Read for dependencies (flutter, bloc, riverpod, etc.) |
| `pom.xml` | Java/Maven | Read for Spring Boot, Java version, dependencies |
| `build.gradle` or `build.gradle.kts` | Java/Kotlin/Gradle | Read for Spring Boot, plugins, dependencies |
| `package.json` | JavaScript/TypeScript | Read for framework (react, next, vue, angular, express, nestjs) |
| `go.mod` | Go | Read for module name and dependencies |
| `Cargo.toml` | Rust | Read for dependencies |
| `requirements.txt` or `pyproject.toml` | Python | Read for framework (django, flask, fastapi) |

If multiple manifest files exist (e.g., monorepo), note all of them and identify the primary one.

Also check:
- `CLAUDE.md` or `.claude/CLAUDE.md` — often contains project identity, conventions, and critical rules
- `README.md` — project description
- `.gitignore` — hints at build tools, IDEs, generated paths
- `docker-compose.yml` or `Dockerfile` — infrastructure hints

## Step 3: Map the Repo Layout

Run `find . -maxdepth 3 -type d` (excluding `node_modules`, `.git`, `build`, `dist`, `.dart_tool`, `target`, `.gradle`) to map the directory structure.

Identify key directories:
- **Source code root** — `src/`, `lib/`, `app/`, `cmd/`
- **Tests** — `test/`, `__tests__/`, `e2e/`, `src/test/`
- **Config** — `src/config/`, `src/main/resources/`, `config/`
- **Docs** — `docs/`, `doc/`
- **API specs** — look for `*.json`, `*.yaml`, `*.yml` files containing "openapi" or "swagger"
- **Generated code** — `generated/`, `build/`, `dist/`
- **Existing PRDs or initiatives** — `docs/initiatives/`, `docs/prd/`

## Step 4: Collect API Documentation (MANDATORY — ask the user)

Do NOT auto-detect API specs. Ask the user directly:

**"Where are your API docs? I need the file paths and/or URLs for every API source this project uses. For example:**
- **Swagger/OpenAPI spec files** (`docs/api.json`, `docs/openapi.yaml`)
- **Curated API registries** (structured markdown listing endpoints)
- **External URLs** (hosted Swagger UI, Postman collections)
- **Code annotations** (if endpoints are only documented in source code, tell me which directories)
- **Cross-repo references** (if API contracts live in another repo, give me the local checkout path)

**List as many as you have — I'll verify each one."**

For each source the user provides:
1. **Verify it exists**: if it's a local path, confirm the file is on disk. If it's a URL, note it (don't fetch). If it doesn't exist, tell the user immediately.
2. **Read and classify**: open the file, determine its format (OpenAPI 3.x, Swagger 2.0, structured markdown, code annotations, etc.) and what it covers (which service, which endpoints).
3. **Summarize**: note how many endpoints it documents, which services/domains it covers, and any gaps you notice.

After verifying all sources, write `docs/api-sources.md` — a consolidated reference document:

```markdown
# API Documentation Sources

> Auto-generated by project-setup. The researcher and PRD agents use this file
> to find API contracts. Update this file when API sources change.

## Sources

| # | Source | Format | Path / URL | Covers | Endpoints |
|---|--------|--------|-----------|--------|-----------|
| 1 | [name] | [OpenAPI 3.0 / Swagger 2.0 / API registry / code annotations] | `[path]` | [which service/domain] | [count or "~N"] |

## Verification Method

[How to confirm an endpoint exists — derived from the sources above. E.g., "Look up the endpoint in `docs/api-registry.md` by path + method. For endpoints not in the registry, check code annotations in `src/controllers/`."]

## Gaps

[Any services or endpoints the user mentioned that aren't covered by these sources.]
```

Save the path `docs/api-sources.md` into project-context.md under **API Documentation → Location**.

## Step 5: Extract Domain Knowledge

Read `CLAUDE.md` if it exists — extract:
- Project description and domain
- Business terms and glossary items
- Naming conventions and critical rules
- Any existing agent configuration or pipeline descriptions

Read `README.md` for project description if CLAUDE.md doesn't cover it.

## Step 6: Detect Conventions

From the files already read (CLAUDE.md, README, source files, config), extract:
- **File naming**: look at existing source files on disk — kebab-case, snake_case, PascalCase, camelCase
- **Branch naming**: ask the user, or infer from CLAUDE.md / contributing docs if they exist
- **Commit style**: ask the user, or infer from CLAUDE.md / contributing docs if they exist
- **Test location**: check if tests are co-located or in a separate tree

## Step 7: Recommend Section Packs

Based on the detected project type, recommend section packs:

**Frontend web app** (React, Vue, Angular, Next.js):
- screen-flow, navigation, analytics-events, localization, component-mapping, feature-flags, accessibility, responsive-layout

**Mobile app** (Flutter, React Native, Swift, Kotlin):
- user-journey, screen-flow, navigation, analytics-events, localization, component-mapping, feature-flags, accessibility, platform-considerations

**Web app porting from mobile** (detected by mobile research docs, mobile references in CLAUDE.md):
- all frontend packs + create a custom `mobile-baseline` pack during Step 8 (document which mobile behaviors to match/change/skip, source files, API endpoints used by mobile)

**Projects with a design system / Storybook / prototype components** (detected by Storybook config, `design-system/` or `__prototype__/` directories, or component library with visual examples):
- add design-prototype to whichever list above applies

**Backend service** (Spring Boot, Express, NestJS, Django, FastAPI, Go):
- feature-flags, database-changes, service-integration, monitoring

**Fintech / regulated** (detected by KYC, compliance, AML references in code or docs):
- add compliance to whichever list above applies

## Step 8: Discuss Custom Section Packs and Research Steps (MANDATORY — ask the user)

After recommending built-in packs, check if the project has patterns that aren't covered. Look for signals:
- Central tracking catalogs (discrepancy logs, decision logs, migration checklists)
- Project-specific documentation patterns visible in existing PRDs or docs
- Workflow artifacts mentioned in CLAUDE.md (e.g., mock data strategy, prototype requirements)
- Sibling repos, external API registries, or other external sources referenced in docs

Then ask the user TWO questions:

**Question 1 — Custom PRD Sections:**

**"The built-in section packs cover [list what's enabled]. Does your project have any recurring PRD sections or patterns that aren't covered by these? For example:**
- **A discrepancy/diff tracker** (e.g., documenting differences between platforms or versions)
- **A mock data or test data section** (e.g., what MSW handlers or fixtures to create)
- **A migration or rollout plan** (e.g., feature rollout steps, data migration checklists)
- **Any other section your team always includes in specs**

**If yes, I'll help you create custom section packs for them."**

**Question 2 — Custom Research Steps:**

**"Does your research process require any steps beyond scanning this repo? For example:**
- **Checking a sibling repo** (e.g., a mobile app whose API contracts this project ports)
- **Querying an external API registry or documentation source**
- **Reading a compliance database or regulatory reference**
- **Any other external source the researcher should check**

**If yes, I'll create research step files that the researcher agent executes automatically."**

If the user identifies custom packs:
1. **Check if a built-in pack already covers the same domain.** If so, edit the built-in pack file in the section packs directory to add the project-specific details (Storybook paths, issue workflows, authority order, etc.) rather than creating a second file. Enable it (`[x]`) in the Included Section Packs list. Do NOT create a custom pack that duplicates a built-in one.
2. For genuinely new concerns with no built-in equivalent, create a `.md` file in the section packs directory, following the same format as built-in packs:
   - A title line (`# Section Pack: [Name]`)
   - An `Insert into` tag specifying where it goes in the PRD (Context, Contract, Technical, or Boundaries)
   - A `When` tag specifying when to include it
   - A `GUIDE` block explaining what to write and why
   - A template with placeholder content
3. Add custom pack file paths to the "Custom Section Packs" list in project-context.md
4. Explain: "The prd-writer will read these and include them in every PRD. The prd-reviewer will verify they're filled in. You can edit the pack files anytime to adjust the template."

If the user says no custom packs are needed, set "Custom Section Packs" to `- none`.

If the user identifies custom research steps:
1. Create a `.md` file in `docs/research-steps/` (create the directory if it doesn't exist), following this format:
   ```markdown
   # Research Step: [Name]

   ## When
   [Always, or a condition — e.g., "only when initiative touches API endpoints"]

   ## What To Do
   1. [Specific instruction]
   2. [Next instruction]

   ## Where To Look
   - [Path, repo, or URL to check]

   ## Output Section
   Add a `## [Section Name]` section to the research doc with: [what to include].
   ```
2. Add the file path as a checked item under "Custom Research Steps" in project-context.md
3. Explain: "The researcher agent will execute these steps after scanning this repo and include the results in every research document. You can edit the step files anytime."

If the user says no custom research steps are needed, leave "Custom Research Steps" as `- none` and move on.

## Step 9: Determine Research Configuration

Based on project type, fill in:
- **Knowledge base**: check for `.ai-docs/`, `docs/knowledge-base/`, or similar
- **Entry points**: derive from project type (controllers, pages, screens, routes)
- **API tracing method**: derive from project type and framework
- **Permalink format**: read `git remote get-url origin` and construct the pattern (GitHub, GitLab, Bitbucket, or "none")

Research scope:
- **Include**: source directories identified in Step 3
- **Exclude**: test fixtures, generated code, build output, node_modules, other apps in monorepo

## Step 10: Draft project-context.md

Fill in `.claude/project-context.md` with everything you've gathered:

1. Replace all placeholder values with actual project data
2. For anything you couldn't determine, use `[TODO: description of what's needed]` markers
3. **Remove all `> GUIDE` blocks** — they are instructions for you, not for agents reading the file later
4. **Remove the template header** (`> Fill this file once per project...`) — it's scaffolding, not content
5. Mark recommended section packs as `[x]` and others as `[ ]`

The finished file should read as a clean reference document, not a partially-filled template.

## Step 11: Present to User

Show the user a summary of what you detected and drafted:

```
Project: {name}
Type: {detected type} ({framework} {version})
API docs: {location or "none found"}
Section packs: {list of recommended packs}
TODOs remaining: {count and list}
```

Then say: **"Review `.claude/project-context.md` and fill in any `[TODO]` markers. When you're happy with it, we can delete unused section pack files from `docs/prd-sections/` and run the setup verification."**

## Step 12: Cleanup (after user confirms)

When the user confirms the draft is good:
1. Create an empty `prd-lessons.md` in the project root if it doesn't already exist. This file captures lessons learned across PRD initiatives — the reviewer and writer agents will reference it over time.
2. Keep all section pack files in the section packs directory — even unchecked ones. The user may enable them later for a different initiative. Deleting them forces re-copying from the framework.
3. Run the verification prompt on project-context.md:
   - Confirm no `[TODO]` markers remain
   - Confirm no `> GUIDE` blocks remain
   - Confirm no template scaffolding text remains (e.g., `> Fill this file once per project`)
   - Confirm all checked section pack files exist on disk
   - Confirm all custom research step files exist on disk
   - Confirm `docs/api-sources.md` exists and references valid files
   - Confirm `prd-lessons.md` exists in the project root
4. Report: "Setup complete. You can now run `/create-prd {initiative}` or individual agents."
